## Phases

### Phase 1: Generate the brownfield fixture

Run Clarifier evolution mode on CashPulse with a single change request. Capture real outputs — no fabrication.

**Recommended change request:** "Add recurring transactions — let users mark a transaction as recurring (weekly/monthly/yearly) and see upcoming recurrence on the dashboard."

This choice is defended on three grounds: (a) it touches an existing screen (dashboard) and adds a new entity field (`recurrence` on Expense) and likely a new screen (recurrence management), so the change classification spans `ui + component + data-model` axes — a useful multi-axis test. (b) It's plausible as a real-user change request, not a contrived edge case. (c) It produces non-trivial deltas on the existing dashboard DesignSpec without rewriting it.

**Inputs:** existing `agentforge/designs/*.json` from M0's CashPulse baseline run; the M0 baseline `EnrichedRequirement`.

**Outputs captured to the fixture:**
- The pre-change DesignSpec set (copy of M0 baseline)
- The post-change `EnrichedRequirement` (Clarifier evolution-mode output)
- The pre-change PRD (M0 baseline)
- A hand-derived expected `AffectedScreen[]` with node-level annotations — what a human reviewer would identify as affected if running the algorithm by hand

**Script:** `scripts/run-clarifier-cashpulse-brownfield.ts` (one-shot, follows the pattern of `scripts/run-clarifier-cashpulse.ts` from M0 Step 1).
