# 1. Executive Summary

AgentForge is an open-source, community-driven multi-agent framework that orchestrates the complete software development lifecycle (SDLC) from UX design through production deployment and observability. It provides a pluggable, configurable system where specialized AI agents collaborate with human developers across every phase of application development.

The framework is designed to onboard any greenfield application into an agent-assisted workflow where design agents, specification agents, code generation agents, CI/CD agents, and observability agents coordinate through a shared orchestration layer built on the Model Context Protocol (MCP).

**Core thesis:** No single AI agent can build production software. But a coordinated system of specialized agents, each operating within clearly defined boundaries and governed by configurable human-in-the-loop policies, can compress the SDLC from months to days while maintaining quality, security, and architectural integrity.

## 1.1 Key Design Decisions

-   **Open-source (Apache 2.0):** Community-driven development with transparent governance. All core framework code is free and open. Code generated by AgentForge agents is not covered by the AgentForge license and belongs to the user.

-   **Persona B first:** Built for developers on 5-15 person teams who want agents handling the repetitive parts of the SDLC while they focus on architecture and business logic.

-   **Phase 1 stack:** React + Node.js + PostgreSQL. Single supported stack with a pluggable architecture that enables additional stacks as additive extensions, not code changes.

-   **Configurable HITL:** Human-in-the-loop gates are fully configurable per team, per app, and per phase. Teams choose their autonomy level across four policy tiers.

-   **LLM-agnostic:** Pluggable provider layer supporting Claude, GPT, Gemini, and local models. Each agent can use a different provider optimized for its task. Providers support both request/response and streaming modes.

-   **Messaging-first UX:** Slack (interactive, primary) and Telegram (interactive, secondary) as HITL channels with live-updating task boards. CLI as universal fallback. Dashboard deferred to Phase 2.

-   **GitHub Actions sandbox:** Agent-generated code executes in isolated CI environments, never on the developer\'s machine. Results are validated before any PR is created.

## 1.2 Competitive Positioning

AgentForge\'s real competitor is not CrewAI or AutoGen. It is a senior engineer with Claude Code and good taste who can already do spec-driven development, code generation, automated testing, and CI/CD deployment with existing tools in 2 hours.

AgentForge exists because that ad-hoc setup does not scale to teams. It does not provide repeatable workflows, configurable governance, cost controls, audit trails, or progressive trust models. A solo setup works until you need a second developer to trust the same agents with the same guardrails.

**The positioning statement:** AgentForge gives developer teams repeatable, governed, cost-controlled agent workflows across the full SDLC that no combination of individual tools can provide out of the box.

# 2. User Personas

## 2.1 Primary Persona: The Force Multiplier (Persona B)

**Who:** A developer (SE3-SE5 level) on a 5-15 person team who wants to use AgentForge to handle the parts of the SDLC that slow them down: boilerplate, test writing, PR creation, CI debugging, design-to-code translation.

**Pain points:** Context-switching between 8-12 tools. Manual stitching of design-to-code pipelines. Writing repetitive CRUD endpoints and tests. Waiting for CI feedback loops. No standardized way to configure agent autonomy levels across a team.

**What they want:** Agents handling the boring parts while they focus on architecture and business logic. Integration with existing team workflows (Slack, GitHub, Jira). Configurable approval gates so the team lead controls what agents can do autonomously.

**Default HITL:** Design and deploy at full_approval. Code generation at review_and_override. Tests at notify_only.

## 2.2 Secondary Persona: The Indie Builder (Persona A)

**Who:** A solo developer (3-7 years experience) building a SaaS product alone. They don\'t have a team to delegate to. They want AgentForge to be their team.

**Pain points:** Building everything alone is slow. Context-switching between design, frontend, backend, DevOps, and testing. Paying out of pocket for tools. Wanting to ship an MVP in days, not weeks.

**What they want:** Maximum autonomy. Run agentforge init on Saturday, have a deployed MVP by Sunday night. Notifications on their phone via Telegram so they can approve from anywhere.

**Default HITL:** Everything at notify_only or review_and_override. Only production deploy at full_approval.

# 3. Problem Statement

The current landscape of AI-assisted development is fragmented. Individual tools excel in isolation but no framework connects them into a coherent, end-to-end pipeline. This creates three critical gaps.

## 3.1 The Glue Problem

Teams must manually stitch together 8-12 tools to get from idea to production. The integration burden falls on senior engineers who should be solving domain problems, not plumbing agent infrastructure.

## 3.2 The Context Loss Problem

When design decisions move from Figma to a specification document to code generation, each handoff loses context. The code generation agent does not know why the designer chose a specific layout. The CI/CD agent does not know which components are experimental versus production-ready.

## 3.3 The Governance Problem

There is no standardized way to configure how much autonomy each agent has. A startup might want fully autonomous deployment, while an enterprise bank needs human approval at every gate. Current tools force a one-size-fits-all model.

# 4. Framework Architecture

AgentForge is structured as a layered architecture with clear separation of concerns. Each layer is independently replaceable and extensible.

## 4.1 Architecture Layers

  --------------- ---------------------------------------------------------------- ------------------------------------------
  **Layer**       **Responsibility**                                               **Key Technologies**

  Orchestration   Supervisor agent, workflow engine, state management, event bus   LangGraph (Python) / custom DAG engine

  Agent Runtime   Agent lifecycle, memory, tool access, LLM routing, streaming     MCP protocol, provider adapters

  Integration     Adapters for external tools (Figma, GitHub, Slack, Telegram)     MCP servers, REST/GraphQL adapters

  Governance      HITL gates, approval workflows, audit logging, cost controls     Middleware pattern, policy-as-code, RBAC

  Observability   Tracing, metrics, eval, drift detection                          OpenTelemetry, Langfuse/OpenLIT adapters
  --------------- ---------------------------------------------------------------- ------------------------------------------

## 4.2 Core Design Principles

-   **Spec-as-source-of-truth:** Every application generates a living specification (YAML) that evolves with the project. All agents read from and write to this spec. It is version-controlled alongside code. Post human-approval, code becomes truth and the spec auto-syncs to match (see Section 8: Context Resolution).

-   **Agent isolation:** Each agent operates in a sandboxed context with explicit permissions. A design agent cannot push to production. A CI/CD agent cannot modify design files. Permissions are defined in the agent contract and enforced by the governance middleware.

-   **Event-driven coordination:** Agents communicate through an event bus, not direct calls. When the design agent completes a page, it emits a DesignComplete event. The spec agent subscribes and begins work. No agent ever calls another agent directly (see Section 7: Agent Communication Protocol).

-   **Fail-safe defaults:** Every destructive action (merge, deploy, delete) requires explicit human approval by default. Teams opt into autonomy, not out of safety.

-   **Cost-aware execution:** Every agent call tracks token usage, API cost, and wall-clock time. Budget limits are enforced at three levels: per-agent, per-phase, and per-project. Runaway loops are automatically terminated via circuit breakers.

## 4.3 Process Architecture

For Phase 1, the entire framework runs as a single Node.js process invoked via the CLI. The architecture uses TypeScript interfaces internally, with the orchestration engine (LangGraph, Python) communicating via a local REST/gRPC bridge.

> agentforge CLI (TypeScript / Commander.js)
>
> \|\-- calls \--\>
>
> agentforge-engine (Python)
>
> \|\-- Orchestrator (LangGraph graphs per phase)
>
> \|\-- Agent Runtime (provider routing, sandboxing, budget)
>
> \|\-- Event Bus adapter (in-memory for v1, Redis Streams later)
>
> \|\-- MCP Client layer (Figma, GitHub, Slack, Telegram)
>
> \|\-- Governance engine (middleware, not a separate service)

## 4.4 API Contracts Between Layers

Within a single process, layers communicate via TypeScript interfaces. The governance layer is implemented as middleware that wraps every agent action, not as a separate service.

> Orchestrator Interface:
>
> startPhase(phase, config) -\> void
>
> getStatus() -\> ProjectState
>
> pausePhase(phase) -\> void
>
> approveGate(gateId, decision) -\> void
>
> Agent Runtime Interface:
>
> executeAgent(agentContract, context) -\> Result
>
> getAgentStatus(agentId) -\> AgentState
>
> Governance Interface (middleware):
>
> checkPermission(agent, action) -\> Allow \| Deny
>
> enforceHITL(action) -\> Proceed \| Pause \| Notify
>
> checkBudget(agent, estimatedCost) -\> Allow \| Deny
>
> MCP Client Interface:
>
> callTool(server, method, params) -\> Result
>
> Event Bus Interface:
>
> emit(event) -\> void
>
> subscribe(eventType, handler) -\> void

Every executeAgent call passes through checkPermission, checkBudget, and enforceHITL before the agent\'s LLM call fires. If any check fails, the action is blocked before spending tokens.

# 6. Schema Versioning

Every YAML file includes a version field. The agentforge migrate command reads file versions, applies pending transforms, and updates files. Even if v1.0 to v1.1 has zero migrations, the infrastructure exists from day one.

> agentforge migrate \# applies pending migrations
>
> agentforge migrate \--dry \# shows changes without applying

Migration functions transform one schema version to the next. New fields get sensible defaults. Removed fields are archived. The migration runner is idempotent and can be run multiple times safely.

# 7. Agent Communication Protocol

Agents do not communicate with each other directly. All coordination flows through the event bus and shared spec files. This is a deliberate architectural decision.

## 7.1 Inter-Phase Communication

When an agent completes its work, it emits an event to the event bus with a pointer to the updated spec file. Downstream agents subscribe to that event type, read the updated spec, and begin their work. The orchestrator manages sequencing and dependency resolution.

> Design agent completes wireframe
>
> -\> emits DesignPhaseComplete { spec_ref: \"components/dashboard.yaml\" }
>
> -\> Spec agent subscribes, reads updated spec, generates component specs
>
> -\> emits SpecComplete { spec_ref: \"components/dashboard.yaml\" }
>
> -\> Code agent subscribes, reads spec, generates component code

## 7.2 Intra-Phase Communication

For agents within the same phase running in parallel (e.g., frontend coder and backend coder), coordination happens through the task dependency graph and shared spec files. If the backend agent needs something the frontend agent produced, the dependency is declared in the task graph. The orchestrator will not start the backend task until the frontend task\'s output is committed and the spec is synced.

## 7.3 Why Not Direct Agent-to-Agent Messaging

Direct messaging between agents creates a debugging nightmare. With event bus plus shared files, every interaction is logged, version-controlled, and inspectable. If agent B produces wrong output, you can trace back to the exact spec version it read and the exact event that triggered it. With direct messaging, that audit trail does not exist.

# 8. Context Resolution and Spec Sync

The spec is the intent. Code is the implementation. When they diverge after human approval, code wins and the spec auto-syncs.

## 8.1 Resolution Lifecycle

1.  Spec is the source of truth during planning and task assignment.

2.  Agent generates code from the spec. Code may deviate (extra props, renamed fields, structural improvements).

3.  Human reviews the PR. If approved, the deviation is intentional.

4.  Post-merge: a lightweight spec sync agent diffs the spec against merged code and updates the spec to match reality.

5.  Spec is truth again for the next task, now reflecting the approved implementation.

## 8.2 Drift Detection

The spec sync agent flags significant deviations to the human rather than silently updating. New endpoints, changed data models, and removed fields require explicit human acknowledgment. Minor additions (extra props, utility functions) are auto-synced with a commit noting the change.

## 8.3 Conflict Rule

**Human always wins.** If a human edits the spec while an agent is also writing to it, the agent discards its changes and re-reads the human\'s version. File locking during agent writes prevents concurrent corruption, but human edits detected mid-agent-write take priority unconditionally.

# 11. SDLC Phase Details

## 11.1 Phase 1: Design Agents

The design phase operates as a collaborative loop between human designers and AI agents. Figma changes are the source of truth for the design phase.

### 11.1.1 Design Loop Workflow

10. User requests a page via natural language or edits directly in Figma.

11. UX research agent analyzes the request against the spec, identifies data models, suggests layout patterns.

12. Wireframe agent generates a low-fidelity wireframe in Figma with auto-layout.

13. Human reviews in Figma, makes adjustments. Edits are detected by the framework.

14. Visual design agent applies design system tokens to the approved wireframe.

15. Design review agent validates accessibility, responsiveness, and design system compliance.

16. User approves via Slack/Telegram/CLI. Design phase emits DesignPhaseComplete.

### 11.1.2 Figma Integration

AgentForge leverages the Figma MCP server for full bidirectional design-to-code workflow. Read path: get_code extracts design context. Write path: generate_figma_design creates editable Figma layers. Code Connect maps Figma component IDs to actual codebase component paths.

### 11.1.3 Design-Tool Abstraction

While Figma is the primary adapter, AgentForge defines a DesignSurface interface that any design tool can implement: createWorkspace(), readDesign(), writeDesign(), getTokens(), onUserEdit(), lockForAgent(). This enables Framer, Storybook, or code-first workflows in future phases.

## 11.2 Phase 2: Specification and Planning Agents

AgentForge adopts spec-driven development (SDD) as its core paradigm. The spec agent consumes finalized design output and produces: component specs, API specs (OpenAPI 3.1), data model specs, task decomposition, and Architecture Decision Records (ADRs).

The estimation agent analyzes task decomposition and provides effort estimates based on complexity and historical data, enabling sprint planning around agent-assisted development.

## 11.3 Phase 3: Code Generation Agents

### 11.3.1 Multi-Agent Coding Architecture

  ---------------- ------------------------------------------------------------- -------------------------------------
  **Agent**        **Responsibility**                                            **Output**

  Frontend coder   Generates UI components from design spec + Figma context      React components with TypeScript

  Backend coder    Generates API endpoints, business logic, data access layers   Node.js services, Prisma migrations

  Test writer      Generates unit, integration, and e2e tests from specs         Jest, Playwright test suites

  PR reviewer      Reviews generated code for quality, security, architecture    Review comments, requested changes

  Refactorer       Improves existing code without changing behavior              Refactored files with passing tests
  ---------------- ------------------------------------------------------------- -------------------------------------

### 11.3.2 LLM Provider Routing

  -------------------------------- ------------------------------ ---------------------------------
  **Task Type**                    **Recommended Provider**       **Rationale**

  Complex architecture decisions   Claude Opus / GPT-4            Highest reasoning capability

  Standard component generation    Claude Sonnet / GPT-4-mini     Good quality, lower cost

  Test generation                  Claude Sonnet / Gemini Flash   High volume, pattern-based

  Code review comments             Claude Haiku / GPT-4-mini      Fast, cost-effective

  Boilerplate / scaffolding        Local model (Llama/Mistral)    Zero API cost, adequate quality
  -------------------------------- ------------------------------ ---------------------------------

### 11.3.3 Code Generation Workflow

17. Orchestrator distributes decomposed tasks to coding agents based on task type.

18. Each agent receives: relevant spec section, design context via Figma MCP, existing code context, architectural constraints, and agent learnings.

19. Agent generates code in a feature branch following project coding standards.

20. Code is pushed to GitHub. GitHub Actions sandbox runs build and tests.

21. If CI passes, PR is created with description linking to spec and design context.

22. PR reviewer agent performs initial review for quality, security, and architecture compliance.

23. Based on HITL config, PR either requires human approval, auto-merges with notification, or auto-merges silently.

### 11.3.4 Concurrency Model

Each task gets its own agent instance. The orchestrator spawns up to N concurrent agents (configurable via max_concurrent_agents, default 3). When an agent is blocked waiting for CI results, the slot does not open; instead, the next independent task is assigned to a new agent instance up to the concurrency limit. This avoids context-switching within agents while maintaining parallelism.

## 11.4 Phase 4: CI/CD Agents

AgentForge does not replace existing CI/CD infrastructure. It layers agent intelligence on top of the team\'s existing pipeline.

-   **Build agent:** Monitors build failures, analyzes error logs, fixes known patterns automatically or creates tasks with diagnostic context.

-   **Security scanner agent:** Runs SAST/DAST scans on every PR. Assesses severity, generates fix suggestions, auto-fixes low-severity issues or flags for human review.

-   **Deploy agent:** Manages deployment to staging and production. Supports canary, blue-green, and rolling strategies. Always gated by human approval in production by default.

-   **Rollback agent:** Monitors post-deployment health metrics. Initiates automatic rollback if error rates spike above thresholds, with root cause analysis.

## 11.5 Phase 5: Observability Agents

### 11.5.1 Application Observability

-   **Performance monitoring:** Tracks response times, error rates, resource utilization. Anomalies trigger investigation.

-   **Error pattern detection:** Clusters recurring errors, creates bug tickets with diagnostic context linked to code and spec.

-   **User behavior analysis:** Tracks feature usage. Underused features flagged for design reconsideration.

### 11.5.2 Agent Observability

-   **Cost tracking:** Real-time dashboards showing token usage, API costs, wall-clock time per agent, per phase, per project.

-   **Quality scoring:** Code generation scored on test pass rate, review comment density, production incident correlation.

-   **Drift detection:** Monitors agent behavior changes over time. Alerts when quality scores deviate beyond thresholds.

### 11.5.3 Production Observability Boundaries

Observability agents detect and diagnose production issues proactively. They do NOT auto-fix production code. The process is:

24. Observability agent detects anomaly (error rate spike, performance degradation).

25. Agent creates a diagnostic task with logs, stack traces, and affected code paths.

26. Task enters the standard pipeline: spec update, code fix, PR, CI validation, human-approved deploy.

27. The only autonomous production action permitted is rollback to a known-good state, and even that defaults to full_approval HITL.

**Principle:** The proactive part is detection and diagnosis. The fix always goes through the governed pipeline. An agent autonomously pushing code to production to fix something it detected is how you get outages.

### 11.5.4 Feedback-to-Design Loop

When the observability layer detects a production issue, it can create a feedback event that triggers the design phase for the affected feature: Production issue detected, spec agent creates improvement task, design agent proposes changes, code agent implements, CI/CD deploys fix, observability confirms resolution.

# 12. Research and Continuous Improvement Agents (Phase 3)

Research agents periodically analyze the deployed application and suggest improvements that feed back into the design phase. They are not part of Phase 1 because the required plumbing (event bus, task creation, HITL approval) will already exist. Adding research agents later is a new agent contract plus prompts, not an architecture change.

## 12.1 Research Agent Types

-   **UX research agent:** Crawls the deployed app, analyzes user flow patterns, identifies friction points, suggests UX improvements.

-   **Performance analyst:** Runs Lighthouse audits, identifies bundle size issues, suggests code-splitting opportunities, flags slow API endpoints.

-   **SEO auditor:** Checks meta tags, structured data, page speed, mobile responsiveness, and search ranking factors.

-   **Accessibility auditor:** Validates WCAG compliance, identifies contrast issues, missing ARIA labels, keyboard navigation gaps.

## 12.2 Research Workflow

Research agents subscribe to a scheduled cron event (configurable cadence: weekly, bi-weekly, monthly). Each agent runs its analysis, produces a findings report, and emits ImprovementSuggested events. These create tasks in the backlog with suggest_only HITL status, meaning the human reviews and decides whether to action them. Research agents never trigger code changes directly.

# 13. Human-in-the-Loop Configuration

## 13.1 HITL Policy Levels

  ------------------- -------------------------------------------------------- --------------------------------------------------
  **Level**           **Behavior**                                             **Typical Use**

  Full approval       Agent proposes, human must approve before any action     Regulated industries, production deploys, design

  Review + override   Agent acts, human can override within a time window      Code reviews, design iterations

  Notify only         Agent acts autonomously, human notified after the fact   Test generation, linting, doc updates

  Fully autonomous    Agent acts without notification. Audit log available.    CI builds, metrics collection
  ------------------- -------------------------------------------------------- --------------------------------------------------

## 13.2 Progressive Trust Model

Agent autonomy can increase over time based on performance. If the code generation agent\'s PRs have been approved without changes for 20 consecutive PRs, the team can configure automatic escalation from full_approval to review_and_override. This is opt-in and fully configurable.

## 13.3 Escalation Policy

When a human does not respond within the configured timeout (default: 60 minutes), the framework pauses all dependent tasks and sends escalation to the secondary channel. If escalation also times out, the project enters full pause with a stalled notification. The framework never auto-approves a gated action on timeout.

# 14. Messaging Integration and Task Management

Slack and Telegram are the primary developer interfaces for Phase 1. The CLI serves as a universal fallback. The dashboard is deferred to Phase 2.

## 14.1 Messaging Abstraction Layer

All messaging is abstracted behind a two-layer interface. Layer 1 (HITLChannel) defines the core contract every channel must implement: sendNotification, requestApproval, onDecision, updateStatus. Layer 2 (RichHITLChannel) adds optional capabilities: sendTaskBoard (live-updating task list), sendCodePreview, startThread.

Slack implements both layers. Telegram implements core plus partial rich capabilities. CLI implements core only. The orchestrator checks channel capabilities and sends richer content where supported.

## 14.2 Channel Routing

Approval requests are sent to all configured channels. The first response from any channel wins. This means the developer can approve from their laptop (Slack) or phone (Telegram) without configuring per-channel routing.

Status updates go to the primary channel only. Critical alerts go to all channels.

# 15. Code Sandbox Strategy

Agent-generated code never executes on the developer\'s machine. GitHub Actions serves as the isolated sandbox environment for Phase 1.

## 15.1 Sandbox Workflow

28. Agent generates code and pushes to a feature branch.

29. Push triggers a GitHub Actions workflow that installs dependencies, runs build, runs tests.

30. If CI passes, the agent creates a PR with full context.

31. If CI fails, the CI agent captures logs, sends them to the coding agent with diagnostic context, and the coding agent fixes in the same branch.

32. Maximum 3 CI retry cycles per task before escalating to human.

## 15.2 Latency Mitigation

GitHub Actions has a 2-5 minute cold start per cycle. To mitigate perceived slowness, the orchestrator runs up to max_concurrent_agents tasks in parallel. When one agent is blocked waiting for CI, other agents continue working on independent tasks. The developer sees multiple tasks progressing simultaneously rather than one slow task.

## 15.3 Kill Switch

The agentforge abort command provides granular control over runaway agents:

> agentforge abort \<task_id\> \# stop agent, leave branch
>
> agentforge abort \<task_id\> \--cleanup \# stop agent, delete branch
>
> agentforge abort \--all \# emergency stop everything

When abort is issued: the task status is set to aborting, the current LLM call completes (tokens are consumed), the agent runtime checks status and stops, the branch is preserved for inspection (unless \--cleanup is specified), and the task status moves to aborted. All channels are notified.

# 17. LLM Provider Abstraction

The provider layer supports both request/response and streaming modes from day one.

## 17.1 Provider Interface

> LLMProvider:
>
> complete(prompt, options) -\> CompletionResult
>
> stream(prompt, options) -\> AsyncIterable\<StreamChunk\>
>
> StreamChunk:
>
> { type: \'token\', content: string, token_count: number }
>
> { type: \'tool_call\', name: string, args: object }
>

Streaming enables real-time CLI output (code appears as it\'s generated), periodic Slack progress updates during long tasks, and real-time budget enforcement (abort mid-generation if approaching limits).

Each agent contract specifies whether it uses streaming or request/response mode. Code generation agents use streaming for progress visibility. Lightweight agents (spec sync, validation) use request/response for simplicity.

# 19. Security and Governance

## 19.1 Agent Permissions Model

AgentForge implements a least-privilege model. Each agent is granted minimum permissions for its role, defined in the agent contract and enforced by governance middleware.

-   **Read permissions:** Which resources the agent can read (spec, design files, code, environment variables).

-   **Write permissions:** Which resources the agent can modify (design files, code branches, CI config).

-   **Execute permissions:** Which actions the agent can trigger (deploy, merge, rollback).

-   **Denied permissions:** Explicit denials that override inherited permissions. A design agent is always denied deploy permissions.

## 19.2 Secret Management

Agent credentials are managed through vault integration. Agents never see raw secrets. They receive scoped, time-limited tokens that are automatically rotated.

## 19.3 Audit Logging

Every agent action is logged to an immutable audit trail: agent identity, action taken, input context, output produced, approving human, cost incurred, and timestamp. The audit trail is queryable and exportable for compliance.

## 19.4 Cost Governance

  ------------- ----------------------------------------- ------------------------------------------------
  **Level**     **Scope**                                 **Behavior**

  Per-agent     Maximum tokens/cost per individual task   Prevents runaway loops via circuit breaker

  Per-phase     Maximum cost for an entire SDLC phase     Alerts at 80%, pauses at limit

  Per-project   Monthly or quarterly budget cap           Hard stop with human authorization to continue
  ------------- ----------------------------------------- ------------------------------------------------

## 19.5 Generated Code Licensing

Code generated by AgentForge agents is not covered by the AgentForge Apache 2.0 license and belongs entirely to the user. This is explicitly stated in the LICENSE file and documentation to prevent concerns from teams building proprietary software.

# 20. Failure Modes and Recovery

## 20.1 Critical Failures (Phase 1)

  -------- ------------------------------------- ---------------------------------------------------------------------------------------------------------------------------------------
  **ID**   **Failure Mode**                      **Recovery**

  F1       LLM returns malformed/garbage code    Self-test catches it. Retry with error context (max 3). If all fail, create needs-human task, notify via Slack/Telegram.

  F2       LLM rate limit or API outage          Exponential backoff. Failover to secondary provider if configured. Pause phase + notify human.

  F3       Budget exceeded mid-task              Hard stop. Do not commit partial work. Notify human with cost breakdown. Human increases budget or takes over.

  F4       HITL approval timeout                 Pause dependent tasks. Escalate to secondary channel. If escalation times out, full pause + stalled notification. Never auto-approve.

  F5       Git merge conflict between agents     Orchestrator detects before merge. Second agent rebases. If auto-rebase fails, create resolve-conflict task for human.

  F6       CI pipeline fails on generated code   CI agent captures logs, sends to coding agent. Coding agent fixes in same branch, re-triggers CI. Max 3 cycles, then escalate.

  F10      Slack/Telegram API failure            Retry. Fall back to secondary channel. If all fail, CLI polling mode. Ugly but functional.

  F11      Agent stuck in loop                   Circuit breaker: \>5 LLM calls without state change triggers force-stop. Log context. Notify human.
  -------- ------------------------------------- ---------------------------------------------------------------------------------------------------------------------------------------

## 20.2 High Severity Failures (Phase 2+)

  -------- ---------------------------------------- -----------------------------------------------------------------------------------------------------------------------------
  **ID**   **Failure Mode**                         **Recovery**

  F7       Figma MCP server unavailable             Retry 3x. Fall back to code-first design (Storybook). Notify human of degraded state.

  F8       Architecturally wrong but passing code   PR reviewer catches violations. If missed, human review catches it. HITL full_approval for architecture code.

  F9       Spec is ambiguous or contradictory       Spec agent flags ambiguity. Creates clarification task. Blocks dependent code gen until resolved.

  F12      Concurrent spec edits (human + agent)    File locking during agent writes. Human edit detected mid-write: agent discards, re-reads human version. Human always wins.

  F13      Design token mismatch                    Design agent validates mapping before completing. Unknown tokens flagged for human.

  F14      GitHub Actions runner quota exhausted    Queue with backoff. Notify of delay. Suggest self-hosted runners if recurring.

  F15      MCP server returns unexpected schema     Adapter validates response. On failure: log warning, fall back to cached behavior, notify human.
  -------- ---------------------------------------- -----------------------------------------------------------------------------------------------------------------------------

# 21. Developer Experience Narrative

This section describes the moment-to-moment experience of using AgentForge, answering the question: what does a Tuesday morning look like?

## 21.1 First Run (One-Time Setup)

The developer installs the CLI (npm install -g agentforge), runs agentforge init, and answers 5 questions. In under 3 minutes, the project is scaffolded, agents are registered, and Slack/Telegram channels are connected. The developer runs agentforge start design.

## 21.2 Design Phase

The developer describes the desired page in natural language via the CLI (`agentforge design "A dashboard with revenue chart and activity feed"`) or by typing in the Slack channel with a "design:" prefix. After spec approval, the developer gets a Slack message: 6 tasks created for Dashboard page. The live task board updates in real-time. The developer goes to get coffee. ## 21.4 Ongoing Interaction

AgentForge is now tied to the application. The developer uses CLI commands for quick actions and Slack/Telegram for approvals and monitoring. The framework handles the SDLC pipeline autonomously within the configured HITL boundaries. The developer focuses on architecture decisions, business logic, and the parts only a human can do.
